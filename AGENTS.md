## 세션 연속성

사용자가 세션 시작 시 아래와 같은 뉘앙스의 말을 하면, **반드시** `STATE.md` 파일을 읽고 현재 프로젝트 상태를 파악한 뒤 답변하라.

트리거 예시:
- "이전 내용 보고 와"
- "지난 대화 이어서"
- "전에 어디까지 했지"
- "이어서 하자"
- "연속성 유지해줘"
- "전 세션 내용 참고해"
- 그 외 이전 맥락을 요청하는 모든 표현

읽은 후에는 `STATE.md` 기준으로 현재 진행 상태(완료된 것 / 남은 것)를 간략히 요약하고, 이어서 작업을 진행하라.

## 완료 기록 (WORKLOG.md)

이슈 또는 기능 단위 작업이 완료되면 **반드시** `WORKLOG.md`에 기록을 추가하라.

기록 시점:
- 사용자가 "완료", "끝났어", "다음으로 가자" 등 완료 의사를 표시할 때
- PR 생성 직후
- 기능 동작 확인 후

기록 항목:
- 날짜, 이슈 번호, 브랜치명, 상태
- **변경 파일** (생성/수정/삭제 구분, 경로 정확하게)
- **주요 작업 내용** (구현한 기능 요약)
- **트러블슈팅** (발생한 문제 + 원인 + 해결방법) — 있다면
- **부수 결정** (작업 중 내린 판단/컨벤션 결정) — 있다면

신규 진행 중 작업은 `🚧 진행 중` 상태로 미리 stub만 만들어 둬도 OK.

## API 작업 규칙

다음 작업 시작 전 **반드시** `API.md`를 먼저 읽어라:

- Controller, Service, DTO, Repository 코드 작성/수정
- API 엔드포인트(`@GetMapping`, `@PostMapping`, `@PutMapping`, `@PatchMapping`, `@DeleteMapping`) 추가/변경
- Request/Response 구조 변경
- 에러코드(USER_*, AUTH_*) 추가/변경
- 사용자가 "API 만들어줘", "엔드포인트 추가", "회원가입/로그인/마이페이지 구현해줘" 등 언급

읽지 않고 API 코드를 작성하는 것을 금지한다. 명세서와 다른 경로/메서드/필드를 사용하지 말 것.

API 명세서가 변경되면 `API.md`도 함께 업데이트하라 (STATE.md 동기화 규칙과 동일).

## STATE.md 동기화

토큰 소비를 줄이기 위해 변경사항이 생길 때마다 즉시 반영하지 않고, **일정 간격(이슈 하나 완료 시점 또는 대화 흐름상 자연스러운 마무리 지점)** 에 모아서 반영하라.

반영 대상:
- 도메인 설계 변경 (ERD, 엔티티 필드, 역할 등)
- 정책 변경 (비즈니스 규칙 등)
- 액션 아이템 완료 처리 (`[ ]` → `[x]`)
- 마일스톤 진행 상황
- 기타 설계 결정 사항

> API 엔드포인트/파라미터/응답/에러코드 등 명세 변경은 `API.md`에 반영하라.

반영 시 지켜야 할 것:
- 사용자에게 "반영했습니다" 라고 별도 고지 없이 **조용히 처리**하라
- 기존 포맷과 스타일을 유지하라
- 변경 일자가 있는 경우 `⚠️ 변경사항` 섹션에 날짜와 함께 추가하라

### 0. 기본 톤

- **한국어 친근체** 사용 (~예요, ~해요, ~하시면 돼요)
- 적절한 **이모지** 사용 (남발 X, 강조 위주: ✅ ⚠️ 🎉 🔍 ⭐ 📌 🛠️ 등)

- **표 형식**을 적극 활용 (비교/정리/매핑)

- 한 응답에 너무 많은 단계를 묶지 말고, **한 번에 한 단계씩** 진행
- 매 코드 변경 후 **"빌드 확인해주세요!"** 권유하고 응답 기다린 뒤 다음 단계

```markdown
## N️⃣ [작업 제목]

**경로**: `src/main/java/com/example/project/...`

​```java
[코드]
​```

---

|------|------|
| `@Annotation` | 무슨 역할 |
| `메서드명` | 무슨 역할 |

​```java

​```

---

붙여넣고 빌드 확인해주세요!
```

핵심:
- **경로는 항상 명시** (`src/main/java/...` 절대경로 형태)
- 코드 블록은 ```` ```java ```` 사용
- 코드 뒤 **설명 표** 필수 (각 부분의 역할)
- **추가 import** 있으면 별도 섹션
- 끝에 다음 행동 안내 ("빌드 확인", "테스트 해보세요" 등)

### 2. PR 본문 양식

> 🚫 **커밋·PR은 사용자가 직접 한다.** AI 어시스턴트는 `git commit`, `git push`, `gh pr create` 등을 **실행하지 않는다.** 아래 양식대로 **PR 제목 + 본문 내용을 응답으로 제공**하면 사용자가 복사해서 직접 올린다. 커밋 메시지도 마찬가지 — 양식대로 텍스트만 준다. (스테이징/푸시 자동 실행 금지)

> 팀 형상관리자 합의 양식 (2026-05-27 갱신). 체크리스트 + 이슈 번호는 유지, 그 외 군더더기 항목(테스트 결과 섹션 등)은 제거.

#### PR 제목

`[타입] 작업 내용` 형식. 타입은 아래 기준으로 판단:

| 타입 | 사용 시점 |
|------|----------|
| `[Feat]` | 기능 구현, 추가, 생성 |
| `[Update]` | 기존 기능 수정, 보완, 변경 |
| `[Fix]` | 오류, 버그, 예외 수정 |
| `[Ref]` | 리팩토링, 구조 개선, 코드 구조 수정 |

#### PR 본문

````markdown
해당하는 항목에 체크해주세요.

- [x] ⭐ FEATURE
- [ ] 🌱 UPDATE
- [ ] 🐛 BUGFIX
- [ ] ♻️ REFACTOR

---

## 📌 작업한 이슈
- Closes #N

---

## 🛠️ 기능 관련 작업 내용
- 변경 내용 1
- 변경 내용 2
- 변경 내용 3

---

## ♻️ 패키지 변경 사항
- `project/...`: 변경 내용
- `project/...`: 변경 내용
````

#### 작성 규칙

- 체크리스트는 PR 제목의 타입과 동일하게 1개 체크 (`[Feat]` → ⭐ FEATURE 체크).
- 이슈 번호는 `Closes #N` 으로 작성. 다중 이슈는 `Closes #N + Closes #M`.
- **테스트 결과 섹션, 소제목 분리는 작성하지 않는다.** (단순/간결 유지)
- 기능 관련 작업 내용은 실제 변경된 코드 기준 **2~4개 bullet**.
- 패키지 경로는 반드시 `project/...` 형식. 변경 파일을 보고 자동 판단.
- 확인 불가 항목은 "확인 필요"라고 작성.

#### 작성 절차

1. `git status` / `git diff` 로 변경 파일 목록 + 코드 내용 확인
2. 새로 추가되거나 수정된 클래스/메서드를 기준으로 bullet 작성
3. 패키지별로 어떤 작업이 있었는지 간단히 정리

### 4. 작업 단계 분해 양식

### 8. 백로그/완료 기록 자동화

작업 흐름 중 자동으로 처리:

- **백로그 등록 시점**: 사용자가 "나중에 하자", "백로그 등록해줘", 또는 작업 중 발견한 개선점이 현재 PR 범위 벗어날 때
  → `STATE.md` 의 `🔖 백로그 / 나중에 할 일` 섹션에 추가, 등록 날짜 명시
- **완료 기록 시점**: 이슈 단위 작업 완료 시
  → `WORKLOG.md` 에 변경 파일 표 + 트러블슈팅 + 부수 결정 자동 기록
- **처리된 백로그**: 작업 완료 후 백로그에서 `[ ]` → `[x]` 변경

사용자에게 "기록했어요" 라고 별도 고지 없이 **조용히 처리**, 단 어느 파일에 기록했는지 메시지 마지막에 한 줄 언급.

### 10. 마지막 응답에 항상 포함할 것

### ⚠️ 절대 하지 말 것

- ❌ 한 응답에 5개 이상 작업 단계 묶기 (사용자가 따라가기 힘듦)
- ❌ "그냥 알아서 해주세요" 식의 모호한 안내
- ❌ 코드만 던지고 설명 없음
- ❌ 코드블록 안에 ```` ``` ```` 중첩 (raw markdown 깨짐)
- ❌ 영어로 답변 (한국어 친근체 유지)
- ❌ "코드는 위와 같습니다" 같은 무미건조 마무리
- ❌ 백로그/완료 기록 빼먹기
- ❌ **커밋·푸시·PR 직접 실행** (`git commit` / `git push` / `gh pr create` 등) — 사용자가 직접 한다. AI 어시스턴트는 양식대로 내용만 제공
