## 0. 새 채팅용 빠른 요약

- **담당 도메인**: Problems / Submission / Reward Point / Ranking / Badge
- **M3에서 완료한 핵심**: 코드 실행형 문제 풀이, GCS CSV 데이터셋 업로드, Cloud Run Python Runner 실행, 테스트케이스 기반 채점, 정답 포인트 지급, 포인트 랭킹, 문제풀이 화면용 응답 구조 개선
- **M4에서 바로 이어갈 핵심**: 뱃지 기능 실제 구현, 강의-문제/강좌-문제세트 연동 API, GCS 보안 강화
- **중요한 현재 상태**: 뱃지 API 경로는 존재하지만 서비스, 엔티티, 저장소, GCS 이미지 저장, 에러코드는 대부분 스텁이다. 강의-문제 연동 API는 초안 상태이며 구현 전 Course/Lecture 담당 도메인과 협의가 필요하다.
- **주의**: `Badge`는 “구조만 있음” 상태이므로 API가 있다고 완료로 보면 안 된다.

## [2026-05-31] Problems 도메인 포트 기반 의존 구조 개선 ✅

| PR `#325`, `#328` | ref/problem 병합 |

### 주요 작업 내용

- 하위 도메인 어댑터가 다른 하위 도메인의 Spring Data Repository를 직접 의존하던 구조를 줄였다.
- `ProblemDataset`, `ProblemTestCase`, `Execution`, `ProblemStartCode`, `Submission`, `Progress` 쪽 의존을 Application Port 중심으로 정리했다.
- `ProblemSetCompletedEvent` 흐름을 통해 문제 세트 완료 후 카운트 갱신을 이벤트로 분리했다.

### 왜 이렇게 했나

강사님 피드백은 같은 바운디드 컨텍스트 안이어도 하위 도메인 간 직접 Repository 의존보다 Application Port를 사용하는 쪽이 더 적절하다는 방향이었다. 그래서 저장 어댑터는 자기 저장소에 집중하고, 다른 하위 도메인 조회는 포트를 통해 접근하도록 정리했다.

### 주의할 점

- `ProblemSetResultPersistenceAdapter`는 조회 전용 read adapter 성격이라 아직 여러 Repository를 조합한다. 당장 수정하지 않기로 판단했다.
- 이벤트 기반 완료 카운트는 적용했지만 동시 제출 상황의 중복 증가 방어는 추가 보강 대상이다.

## [2026-05-30] 문제 세트 진입 응답에 전체 소문제 목록 추가 ✅

### 관련 커밋/PR

| 커밋 | 내용 |
|------|------|
| `e1e8680` | 문제 세트 진입 응답에 전체 소문제 목록 추가 |
| PR `#320` | ref/problem 병합 |

### 주요 작업 내용

- `GET /api/v1/problem-sets/{problemSetId}` 응답을 현재 문제 1개 중심에서 화면 단위 응답으로 확장했다.
- 전체 소문제 목록을 `problems[]`로 내려주고, 각 문제의 상태를 `LOCKED`, `UNSOLVED`, `WRONG`, `CORRECT`로 제공했다.
- 이어풀기를 위해 `currentProblemNumber`와 `currentProblemId`를 함께 제공했다.
- 이미 푼 문제의 결과 화면 연결을 위해 `latestSubmissionId`를 제공했다.

### 왜 이렇게 했나

처음에는 “현재 풀 문제 1개”만 내려주면 된다고 봤다. 하지만 프론트 문제풀이 화면에는 사이드바의 전체 소문제 목록, 문제별 상태, 이어풀기, 제출 결과 연결이 필요했다. 그래서 FE가 제출 기록을 직접 해석하지 않도록 BE가 화면 구성에 필요한 상태를 계산해서 내려주는 방향으로 바꿨다.

### 한계

소문제가 매우 많아지면 `problems[]` 응답 크기가 커질 수 있다. 향후에는 목록/상세 API 분리 또는 lazy loading 검토가 필요하다.

## [2026-05-29] 문제 세트 수정 시 데이터셋 교체 지원 ✅

### 관련 커밋/PR

|------|------|
| `d5001b3` | 문제 세트 수정 시 데이터셋 교체 지원 |
| PR `#317` | ref/problem 병합 |

### 주요 작업 내용

- 문제 세트 수정 시 CSV 데이터셋도 함께 교체할 수 있도록 multipart 수정 API를 추가했다.
- 프론트가 파일과 JSON을 함께 보내는 흐름에 맞춰 `multipart/form-data` 소비를 명시했다.

### 트러블슈팅

- **문제**: 프론트에서 `multipart/form-data`로 요청했는데 Spring에서 지원하지 않는 Content-Type 오류가 발생했다.
- **원인**: JSON 요청과 파일 업로드 요청을 같은 방식으로 해석하지 못했고, `@RequestBody` 또는 잘못된 파트 처리 방식이 섞여 있었다.
- **해결**: `consumes = MediaType.MULTIPART_FORM_DATA_VALUE`와 `@RequestPart("request")`, `@RequestPart("datasetFile")` 기준으로 정리했다.

## 3. 중요한 설계 결정 메모

### 3-1. 코드 실행 결과는 `result` 변수로 통일

학생 코드는 최종 답을 `result` 변수에 담는 계약을 사용한다. Python Runner는 마지막에 `print(result)`를 붙여 stdout을 수집한다.

- 장점: 채점 서버가 최종 결과를 일관된 방식으로 받을 수 있다.
- 단점: 문제 설명과 시작 코드에서 `result` 계약을 명확히 안내해야 한다.
- 보완 필요: `result`가 없을 때 사용자 친화적인 에러 메시지 제공.

### 3-2. 코드 실행과 코드 제출은 분리

| 구분 | 역할 |
|------|------|
| 코드 실행 | 학생이 작성한 코드를 실험하고 실행 결과만 확인 |
| 코드 제출 | 테스트케이스 기반 채점, 제출 기록 저장, 진행 상태 갱신, 포인트 이벤트 발행 |

이 분리는 “연습 실행은 학습 상태를 바꾸지 않고, 제출만 평가와 상태 변경을 일으킨다”는 의도를 가진다.

### 3-3. 테스트케이스는 서버가 관리

학생 코드는 서버 테스트케이스와 결합되어 Cloud Run Python Runner에서 실행된다. 테스트케이스별 결과를 저장하기 위해 현재는 테스트케이스마다 실행 요청을 보낸다.

- 장점: 각 테스트케이스의 통과 여부를 개별 저장 가능.
- 한계: 테스트케이스가 많아지면 Cloud Run 호출 횟수도 늘어난다.
- 향후 개선: 여러 테스트케이스를 한 번에 실행하는 배치 실행 구조 검토.

### 3-4. 문제풀이 화면 응답은 화면 단위로 제공

문제 세트 진입 API는 현재 문제 1개만 주지 않고 전체 소문제 목록과 상태를 함께 준다. 이 결정은 FE가 사이드바, 본문, 상태 표시, 결과 조회 연결을 한 번의 응답으로 구성하게 하기 위한 것이다.

### 3-5. 하위 도메인 간 의존은 Application Port로 완화

같은 problems 바운디드 컨텍스트 안이어도 `dataset`, `testcase`, `execution`, `progress`, `submission`이 서로의 Spring Data Repository를 직접 의존하지 않도록 개선했다. 다른 하위 도메인 조회는 포트로 접근하는 방향을 사용한다.

- 운영자 수정 조회 응답에 문제별 테스트케이스 목록을 추가했다.

### 검증

- `gradlew compileJava` 성공
- `gradlew test` 전체 성공

### 부수 결정

- `answer`, `expectedResult`는 실제 채점에 사용되지 않아 API·코드·DB에서 제거하기로 변경했다.
- 새 일괄 등록 채점 기준은 `testCases[].testCode`이며 테스트 순서는 배열 순서로 자동 지정한다.

## [2026-06-09] 문제 정답·기대 결과 필드 제거 및 Swagger 최신화 ✅

### 주요 작업 내용

- 문제 등록·수정·조회와 챗봇·학습·제출 포트에서 문제 정답 `answer` 전달을 제거했다.
- 테스트 케이스 요청·응답·도메인·JPA·채점 포트에서 `expectedResult`를 제거하고 `testCode` 기반 채점으로 통일했다.
- Swagger와 HTTP 테스트 예시를 최신화하고, 사용하지 않는 `PROBLEM_ANSWER_REQUIRED` 에러코드를 제거했다.
- DB 컬럼 제거용 수동 SQL을 추가했다.

### 변경 파일

| 파일 | 변경 |
|------|------|
| `src/main/java/com/wanted/codebombalms/problems/problem/**` | 문제 `answer` 필드와 매핑 제거 |
| `src/main/java/com/wanted/codebombalms/problems/testcase/**` | 테스트 케이스 `expectedResult` 필드와 Swagger 제거 |
| `src/main/java/com/wanted/codebombalms/submission/**` | 채점 포트의 미사용 기대 결과 제거 |
| `src/main/resources/sql/remove-problem-answer-expected-result.sql` | DB 컬럼 제거 SQL 추가 |
| `http/problem-test-case.http` | 최신 요청 예시 반영 |

### 부수 결정

- 정답 조건은 별도 문자열 컬럼이 아니라 실행 가능한 `testCode` 안에서 검증한다.
- DB 마이그레이션 도구가 없어 컬럼 삭제 SQL은 배포 순서와 백업을 확인한 뒤 수동 실행한다.
