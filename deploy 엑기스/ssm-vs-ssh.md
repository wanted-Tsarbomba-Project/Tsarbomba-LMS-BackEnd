# SSM Session Manager vs SSH(.pem) — 접속 방식 비교

> 보안 하드닝(2026-06-24 그릴)에서 **EC2 접속을 `.pem` SSH → SSM Session Manager로 전환**하기로 했다.
> 이 문서는 둘이 정확히 뭐가 다르고, 왜 바꾸는지, 무엇을 잃는지를 정리한다.
> 실행 순서·검증은 핸드오프 `.ai/local/handoff/handoff-2026-06-24.md`, 전체 지도는 [`OVERVIEW.md`](OVERVIEW.md).

---

## 1. 정의

### SSH(.pem) — 지금 쓰는 방식
- **SSH(Secure Shell)**: 원격 서버에 암호화된 셸을 여는 표준 프로토콜. 기본 포트 **22**.
- 인증은 **키페어**로 한다. AWS가 만들어준 `codebomb-ec2-key.pem`(개인키)을 내 PC가 들고, EC2 안에는 짝이 되는 공개키가 박혀 있다.
- 접속 흐름: **내 PC → 인터넷 → EC2의 22번 포트로 직접 인바운드 연결**.
  - 그래서 EC2 보안그룹(SG)에 **22번 인바운드를 열어둬야** 한다. 우리는 IP가 유동(집/학교)이라 `0.0.0.0/0`(전세계)로 열려 있다 ← **공격면**.

```
[내 PC] --(.pem 키)--> 인터넷 --> [EC2 :22 인바운드 열림] 셸
```

### SSM Session Manager — 바꿀 방식
- **SSM = AWS Systems Manager.** EC2/온프렘 서버를 운영·관리하는 AWS 관리형 서비스 묶음이고, 그중 **Session Manager**가 셸 접속 기능이다.
- EC2 안에서 도는 **SSM Agent**(Ubuntu AMI 기본 탑재)가 **바깥쪽(아웃바운드) 443**으로 AWS SSM 엔드포인트에 먼저 연결을 건다. 내가 접속하면 그 통로를 타고 셸이 열린다.
- 인증은 키가 아니라 **IAM 권한**으로 한다. 누가 들어올 수 있는지를 IAM 정책으로 정하고, 모든 접속이 **CloudTrail에 기록**된다.
- 접속 흐름: **인바운드 22 불필요**. EC2는 받는 포트를 0개 열어도 된다.

```
[내 PC/콘솔] --(IAM 권한)--> AWS SSM <--(아웃바운드 443)-- [EC2 SSM Agent] 셸
                                                          ↑ 인바운드 0개
```

**한 줄 차이**: SSH는 *내가 서버 문(22)을 두드린다*, SSM은 *서버가 AWS로 전화를 걸어두고 AWS가 우리를 연결해준다*. 그래서 서버에 열어둘 문이 없다.

## 2. 트레이드오프 비교

| 항목 | SSH(.pem) | SSM Session Manager |
|------|-----------|---------------------|
| **인바운드 포트** | 22 열어야 함 (우리: `0.0.0.0/0`) | **0개** — 닫아도 접속됨 |
| **인증 수단** | `.pem` 개인키 파일 | **IAM 권한**(+선택 MFA) |
| **키 관리** | 파일 공유·분실·유출 위험. 유출 시 누구나 접속 | 키 파일 없음. 권한 회수 = IAM에서 즉시 |
| **누가 들어왔나 감사** | 서버 로그 뒤져야, 사람 식별 약함 | **CloudTrail에 IAM 사용자 단위 기록** |
| **공격면** | 22가 전세계 노출 → 무차별 로그인 시도 표적 | 노출 0 |
| **접속 전제** | SG에 내 IP 허용(유동 IP라 매번 골치) | IP 무관 — 어디서든 됨 |
| **포트포워딩**(9090·RDS 3306 등) | SSH 터널(`-L`)인데 22 열려야 | `AWS-StartPortForwardingSession`으로 인바운드 0개 터널 |
| **팀원 권한 부여** | .pem 파일 나눠줌(회수 불가) | IAM 그룹에 추가/제거로 통제 |
| **의존성** | SSH 클라이언트만 | EC2 IAM 역할 + SSM Agent + (로컬) Session Manager Plugin |
| **콘솔만으로 접속** | 불가 | 가능(브라우저 셸) |

---

## 3. SSM의 장점 (왜 바꾸나)

1. **22번을 완전히 닫는다 → 공격면 제거.** 유동 IP라 SG 화이트리스트가 불가능한 우리 상황에서 `0.0.0.0/0` 22를 닫는 유일한 현실적 방법.
2. **키 파일이 사라진다.** `codebomb-ec2-key.pem` 공유·분실·유출 리스크 제거. 접속 자격을 IAM에서 **즉시 부여/회수**.
3. **감사 추적.** 누가 언제 어느 서버에 들어왔는지 CloudTrail에 IAM 사용자 단위로 남는다.
4. **IP에 안 묶인다.** 집·학교·카페 어디서 접속하든 SG 손댈 필요 없음.
5. **팀원 온보딩이 IAM 한 줄.** `codebomb-developers` 그룹에 넣으면 접속 가능, 빼면 차단. .pem 회수 같은 게 없다.
6. **포트포워딩도 인바운드 0.** Prometheus 9090·RDS 3306을 SG에서 닫고도 로컬에서 터널로 접근.

## 4. SSM의 단점 / 잃는 것 (트레이드오프)

1. **초기 셋업 부담.** EC2 IAM 역할에 `AmazonSSMManagedInstanceCore` 부착 + SSM Agent 동작 + 로컬에 Session Manager Plugin 설치가 필요. SSH는 키 파일 하나면 끝.
2. **AWS·IAM 의존.** AWS SSM 서비스가 죽거나, IAM 권한이 잘못 설정되면 접속 불가. SSH는 AWS 콘솔과 독립적으로 키만 있으면 됨.
3. **아웃바운드 443 필수.** EC2가 SSM 엔드포인트로 나가는 443이 막히면 동작 안 함(우리는 아웃바운드 전체허용이라 OK).
4. **자기 잠금 위험(전환 중).** SSM 접속을 **검증하기 전에** 22를 닫으면 둘 다 막혀 못 들어간다 → 그래서 **실행순서 락**(SSM 검증 → 그 다음 22 닫기)을 둔다.
5. **러닝커브.** 포트포워딩 명령(`aws ssm start-session --document-name AWS-StartPortForwardingSession ...`)이 SSH `-L`보다 길고 낯섦.
6. **CLI 도구 추가.** 로컬 접속엔 AWS CLI + Session Manager Plugin 두 개를 깔아야 함(콘솔 브라우저 셸은 무설치).

---

## 5. 우리 결정

- **접속·포트포워딩을 SSM으로 전면 전환**, EC2 5박스의 **22 인바운드 전부 닫기**. `.pem` 공유는 폐기.
- 단 **안정화까지 `.pem`/22는 남겨두고 SSM을 병행 검증**, 검증(콘솔·로컬 접속 성공)된 *다음에* 22를 닫는다. (자기 잠금 방지 — 핸드오프 §2 실행순서 락)
- 내부 UI(Prometheus 9090·RDS 3306)는 SSM **포트포워딩**으로 접근하고 해당 인바운드도 닫는다. Grafana 3000만 공개 유지(자체 로그인).

> 요약: **단점은 거의 다 "처음 한 번"의 셋업·러닝커브**고, 장점은 운영 내내 누리는 **공격면 제거·키 리스크 제거·감사·팀 권한 통제**다. 그래서 전환한다.
