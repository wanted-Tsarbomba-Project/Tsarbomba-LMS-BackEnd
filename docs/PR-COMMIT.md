# PR · 커밋 양식 (팀 공용)

> 팀 형상관리자 합의 양식. PR 본문/커밋 메시지는 이 형식을 따른다.
> 커밋·푸시·PR·이슈는 **AI가 직접 수행**한다(→ `AGENTS.md` `## Git 워크플로우`). 이 파일은 그때 따를 **양식**을 정의한다.

---

### 2. PR 본문 양식

> AI가 `gh pr create`로 PR을 직접 생성한다. 아래 양식대로 **PR 제목 + 본문**을 채워 올린다(`.github/pull_request_template.md`와 동일 골격). base는 `develop`.

> 팀 형상관리자 합의 양식 (2026-05-27 갱신). 체크리스트 + 이슈 번호는 유지, 그 외 군더더기 항목(테스트 결과 섹션 등)은 제거.

#### PR 제목

`[타입] 작업 내용` 형식. 타입은 아래 기준으로 판단:

| 타입 | 사용 시점 |
|------|----------|
| `[Feat]` | 기능 구현, 추가, 생성 |
| `[Update]` | 기존 기능 수정, 보완, 변경 |
| `[Fix]` | 오류, 버그, 예외 수정 |
| `[Ref]` | 리팩토링, 구조 개선, 코드 구조 수정 |

#### PR 본문

````markdown
해당하는 항목에 체크해주세요.

- [x] ⭐ FEATURE
- [ ] 🌱 UPDATE
- [ ] 🐛 BUGFIX
- [ ] ♻️ REFACTOR

---

## 📌 작업한 이슈
- Closes #N

---

## 🛠️ 기능 관련 작업 내용
- 변경 내용 1
- 변경 내용 2
- 변경 내용 3

---

## ♻️ 패키지 변경 사항
- `project/...`: 변경 내용
- `project/...`: 변경 내용
````

#### 작성 규칙

- 체크리스트는 PR 제목의 타입과 동일하게 1개 체크 (`[Feat]` → ⭐ FEATURE 체크).
- 이슈 번호는 `Closes #N` 으로 작성. 다중 이슈는 `Closes #N + Closes #M`.
- **테스트 결과 섹션, 소제목 분리는 작성하지 않는다.** (단순/간결 유지)
- 기능 관련 작업 내용은 실제 변경된 코드 기준 **2~4개 bullet**.
- 패키지 경로는 반드시 `project/...` 형식. 변경 파일을 보고 자동 판단.
- 확인 불가 항목은 "확인 필요"라고 작성.

#### 작성 절차

1. `git status` / `git diff` 로 변경 파일 목록 + 코드 내용 확인
2. 새로 추가되거나 수정된 클래스/메서드를 기준으로 bullet 작성
3. 패키지별로 어떤 작업이 있었는지 간단히 정리

### 3. 커밋 메시지 양식

```markdown
​```
feat: [한줄 요약] (#이슈번호)

- 변경 사항 1
- 변경 사항 2
- 변경 사항 3
​```
```

prefix 규칙:
- `feat:` — 신규 기능
- `fix:` — 버그 수정
- `refactor:` — 리팩토링
- `test:` — 테스트 추가
- `chore:` — 빌드/설정 등
