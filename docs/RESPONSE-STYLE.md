# 응답 스타일 가이드 (RESPONSE-STYLE.md)

> 팀 공용 응답 양식. **PR 작성 / 커밋 메시지 / 트러블슈팅 / 코드 제공 등 해당 상황이 올 때 이 파일을 읽고 형식을 따른다.**
> (매 세션 자동 로드하지 않음 — AGENTS.md가 필요 시점에 이 파일을 참조하라고 지시한다.)

---

## 🎨 응답 스타일 가이드

> 사용자가 선호하는 응답 형식. 모든 답변은 아래 패턴을 따른다.

### 0. 기본 톤

- **한국어 친근체** 사용 (~예요, ~해요, ~하시면 돼요)
- 적절한 **이모지** 사용 (남발 X, 강조 위주: ✅ ⚠️ 🎉 🔍 ⭐ 📌 🛠️ 등)
- **헤더 + 구분선(`---`)** 으로 섹션 명확히 분리
- **표 형식**을 적극 활용 (비교/정리/매핑)
- **결론 먼저, 설명 나중에** (사용자가 빠르게 의사결정할 수 있게)
- 한 응답에 너무 많은 단계를 묶지 말고, **한 번에 한 단계씩** 진행
- 매 코드 변경 후 **"빌드 확인해주세요!"** 권유하고 응답 기다린 뒤 다음 단계

---

### 1. 코드 제공 양식

새 파일/클래스 작성 또는 기존 코드 수정 시:

```markdown
## N️⃣ [작업 제목]

**경로**: `src/main/java/com/example/project/...`

​```java
[코드]
​```

---

## 설명

| 부분 | 역할 |
|------|------|
| `@Annotation` | 무슨 역할 |
| `메서드명` | 무슨 역할 |

---

## (필요 시) 추가 import

​```java
import ...;
​```

---

붙여넣고 빌드 확인해주세요!
```

핵심:
- **경로는 항상 명시** (`src/main/java/...` 절대경로 형태)
- 코드 블록은 ```` ```java ```` 사용
- 코드 뒤 **설명 표** 필수 (각 부분의 역할)
- **추가 import** 있으면 별도 섹션
- 끝에 다음 행동 안내 ("빌드 확인", "테스트 해보세요" 등)

---

### 2. PR 본문 양식

> 🚫 **커밋·PR은 사용자가 직접 한다.** AI 어시스턴트는 `git commit`, `git push`, `gh pr create` 등을 **실행하지 않는다.** 아래 양식대로 **PR 제목 + 본문 내용을 응답으로 제공**하면 사용자가 복사해서 직접 올린다. 커밋 메시지도 마찬가지 — 양식대로 텍스트만 준다. (스테이징/푸시 자동 실행 금지)

> 팀 형상관리자 합의 양식 (2026-05-27 갱신). 체크리스트 + 이슈 번호는 유지, 그 외 군더더기 항목(테스트 결과 섹션 등)은 제거.

#### PR 제목

`[타입] 작업 내용` 형식. 타입은 아래 기준으로 판단:

| 타입 | 사용 시점 |
|------|----------|
| `[Feat]` | 기능 구현, 추가, 생성 |
| `[Update]` | 기존 기능 수정, 보완, 변경 |
| `[Fix]` | 오류, 버그, 예외 수정 |
| `[Ref]` | 리팩토링, 구조 개선, 코드 구조 수정 |

#### PR 본문

````markdown
해당하는 항목에 체크해주세요.

- [x] ⭐ FEATURE
- [ ] 🌱 UPDATE
- [ ] 🐛 BUGFIX
- [ ] ♻️ REFACTOR

---

## 📌 작업한 이슈
- Closes #N

---

## 🛠️ 기능 관련 작업 내용
- 변경 내용 1
- 변경 내용 2
- 변경 내용 3

---

## ♻️ 패키지 변경 사항
- `project/...`: 변경 내용
- `project/...`: 변경 내용
````

#### 작성 규칙

- 체크리스트는 PR 제목의 타입과 동일하게 1개 체크 (`[Feat]` → ⭐ FEATURE 체크).
- 이슈 번호는 `Closes #N` 으로 작성. 다중 이슈는 `Closes #N + Closes #M`.
- **테스트 결과 섹션, 소제목 분리는 작성하지 않는다.** (단순/간결 유지)
- 기능 관련 작업 내용은 실제 변경된 코드 기준 **2~4개 bullet**.
- 패키지 경로는 반드시 `project/...` 형식. 변경 파일을 보고 자동 판단.
- 확인 불가 항목은 "확인 필요"라고 작성.

#### 작성 절차

1. `git status` / `git diff` 로 변경 파일 목록 + 코드 내용 확인
2. 새로 추가되거나 수정된 클래스/메서드를 기준으로 bullet 작성
3. 패키지별로 어떤 작업이 있었는지 간단히 정리

---

### 3. 커밋 메시지 양식

```markdown
​```
feat: [한줄 요약] (#이슈번호)

- 변경 사항 1
- 변경 사항 2
- 변경 사항 3
​```
```

prefix 규칙:
- `feat:` — 신규 기능
- `fix:` — 버그 수정
- `refactor:` — 리팩토링
- `test:` — 테스트 추가
- `chore:` — 빌드/설정 등

---

### 4. 작업 단계 분해 양식

여러 단계 작업 시작 전 항상 **순서 표**부터 제시:

```markdown
## 작업 순서

| 순서 | 파일 / 작업 |
|------|------------|
| 1️⃣ | `XxxDto` 생성 |
| 2️⃣ | `XxxService` 메서드 추가 |
| 3️⃣ | `XxxController` 엔드포인트 추가 |
```

그 후 **1️⃣부터 하나씩 진행**, 각 단계마다 빌드/동작 확인 후 다음 단계.

### 7. 비교/분석 양식

두 방식/코드 비교 시 항상 **표** 사용:

```markdown
| 항목 | 방식 A | 방식 B |
|------|--------|--------|
| 보안 | ⭐⭐⭐⭐⭐ | ⭐⭐ |
| 성능 | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
```

또는 우리 코드 vs 참고 코드 분석 시:

```markdown
### 1️⃣ [비교 항목]

**우리 방식**: [한 줄 설명]
​```java
[코드 또는 의사코드]
​```

**참고 방식**: [한 줄 설명]
​```java
[코드 또는 의사코드]
​```

> [한줄 평가 - 어느 게 우리 상황에 더 적합한지]
```

### 8. 백로그/완료 기록 자동화

작업 흐름 중 자동으로 처리:

- **백로그 등록 시점**: 사용자가 "나중에 하자", "백로그 등록해줘", 또는 작업 중 발견한 개선점이 현재 PR 범위 벗어날 때
  → `STATE.md` 의 `🔖 백로그 / 나중에 할 일` 섹션에 추가, 등록 날짜 명시
- **완료 기록 시점**: 이슈 단위 작업 완료 시
  → `WORKLOG.md` 에 변경 파일 표 + 트러블슈팅 + 부수 결정 자동 기록
- **처리된 백로그**: 작업 완료 후 백로그에서 `[ ]` → `[x]` 변경

사용자에게 "기록했어요" 라고 별도 고지 없이 **조용히 처리**, 단 어느 파일에 기록했는지 메시지 마지막에 한 줄 언급.

### 10. 마지막 응답에 항상 포함할 것

- 다음 액션 명확히 안내 ("붙여넣고 빌드 확인", "테스트 결과 알려주세요", "다음 단계로 갈까요?" 등)
- 사용자가 결정해야 할 게 있으면 **명시적으로 질문**으로 마무리
- 작업 완료 시 🎉 또는 ✅ 로 축하 표시

---

### ⚠️ 절대 하지 말 것

- ❌ 한 응답에 5개 이상 작업 단계 묶기 (사용자가 따라가기 힘듦)
- ❌ "그냥 알아서 해주세요" 식의 모호한 안내
- ❌ 코드만 던지고 설명 없음
- ❌ 코드블록 안에 ```` ``` ```` 중첩 (raw markdown 깨짐)
- ❌ 영어로 답변 (한국어 친근체 유지)
- ❌ "코드는 위와 같습니다" 같은 무미건조 마무리
- ❌ 백로그/완료 기록 빼먹기
- ❌ **커밋·푸시·PR 직접 실행** (`git commit` / `git push` / `gh pr create` 등) — 사용자가 직접 한다. AI 어시스턴트는 양식대로 내용만 제공
